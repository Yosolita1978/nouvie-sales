### Relaxed customer requirements + enforced "no cédula, no factura"
Two linked changes so informal sales (sin factura) work, while facturas stay tied to a cédula.

**A. Database — `cedula` is now optional**
- `prisma/schema.prisma` — `Customer.cedula` changed `String @unique` → `String?  @unique`. Unique kept; Postgres treats NULLs as distinct, so many cédula-less customers are allowed (no partial index).
- `prisma/migrations/2026-06-12-cedula-optional/migration.sql` (new, manual — this repo uses `db push`, not Prisma Migrate). Ran once against the shared DB via `DIRECT_URL`: `ALTER TABLE customers ALTER COLUMN cedula DROP NOT NULL;` + `UPDATE customers SET cedula = NULL WHERE cedula = '';`.
- Verified before/after: **621 customers intact**, 0 empty strings (nothing to normalize), `is_nullable` NO → **YES**.

**B. Validation — one source of truth (zod)**
- `lib/validation/customer.ts` (new) — `customerCreateSchema` + `customerUpdateSchema`. name/phone required; cedula/email optional with `'' → null` coercion at the boundary; email/cedula formats checked only when a value is present. Helpers `zodErrorMessages` / `zodFieldErrors`.
- Deleted the old inline validation that previously lived in **three** places (the form + both customer routes) so the rules can't drift.

**C. Customer API — null passthrough + race-free duplicates**
- `app/api/customers/route.ts` (POST) and `app/api/customers/[id]/route.ts` (PATCH) — validate via the zod schemas, persist `null` (never `""`), removed the check-then-insert; let the unique index reject duplicates and catch Prisma **P2002** → 409 "Ya existe un cliente con esta cédula."
- PATCH also blocks **clearing** a cédula when the customer already has any order with an `invoiceNumber` (preserves the invariant after the fact).

**D. Form**
- `components/customers/CustomerForm.tsx` — `validate()` now calls the shared schema; cédula/email labels lost their `*` and are marked "(opcional)"; initial data is null-safe (`customer.cedula || ''`).

**E. Factura enforcement (server is authoritative; UI is convenience)**
- `app/api/orders/[id]/pdf/route.tsx` — blocks with 400 when `customer.cedula` is null (the PDF *is* the factura). Local capture also narrows the type so `OrderPdfDocument` keeps `cedula: string`.
- `app/api/orders/[id]/route.ts` (PATCH — the **only** `invoiceNumber` writer) — resolves the effective customer (handles a customer change in a full edit) and blocks assigning a non-empty `invoiceNumber` when there's no cédula. Clearing is always allowed.
- `app/(dashboard)/orders/[id]/page.tsx` — `canInvoice` flag: PDF button disabled + tooltip; "Editar" factura hidden and replaced with the reason when there's no cédula.

## Learned
- This repo has **no `migrations/` folder** — schema changes go through `prisma db push`. For a one-off constraint change I wrote a manual SQL file and ran it with `npx prisma db execute --url "$DIRECT_URL" --file …` instead of `migrate dev` (which could try to reset an un-baselined DB).
- A nullable `@unique` is the correct way to allow many "empty" rows — but only if empties are stored as `NULL`. A single `""` would be unique-distinct; the **second** `""` collides. Hence the `'' → null` coercion in zod and the `UPDATE … WHERE cedula = ''` in the migration.
- The factura invariant has a back-door: editing a customer to clear the cédula *after* a factura exists. Guarded in the customer PATCH (count orders with `invoiceNumber`), not just at issue time.
- Nullable Prisma fields ripple into utility scripts too — `tsc` caught `prisma/backup-orders.ts` (CSV row typed `cedula: string`). Typecheck the whole project, not just `app/`.
